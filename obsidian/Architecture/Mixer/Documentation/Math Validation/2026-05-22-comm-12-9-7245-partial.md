# Partial Verification — B(2,5) comm_12_9 7,245-char reduction

## The claim

The 7,245-char word at `experiments/b25_reduce_core/corrected/final_beam.txt` represents the same element of B(2,5) as the commutator `[c12, c9]`, where `c_k = [[a,b], a^{k-1} b a^{-(k-1)}]`.

Full claim decomposition: see [[2026-05-22-comm-12-9-7245-triage]].

## What has been checked

### Check 1 — Input word matches Havas relator (Sub-claim A)

**What was checked:** The 28,652-char word in `experiments/b25_reduce_core/benchmark_run/words/comm_12_9.txt` was compared byte-for-byte against the `comm_12_9` entry in `experiments/burnside/burnside_bidirectional/b25_relators_havas_verified.txt`.

**Result:** Exact match. Both are 28,652 characters, identical content.

**What this establishes:** The 28,652-char benchmark word is the same string that the Havas verification identified as `comm_12_9`. The Havas file contains 153 relators from the Hall–Sims–Havas computer proof of B(2,5)'s finiteness — these are verified B(2,5) identities (words equal to the identity in B(2,5)).

**Limitation:** This confirms the INPUT to the reduction chain. It does not verify the reduction chain itself.

---

### Check 2 — Abelianization (0,0) preserved at all stages (necessary condition)

**What was checked:** The abelianization map φ: B(2,5) → Z/5 × Z/5 defined by a ↦ (1,0), b ↦ (0,1), A ↦ (-1,0), B ↦ (0,-1). Computed (a-exp mod 5, b-exp mod 5) for all stage outputs as reported in `experiments/b25_reduce_core/corrected/README.md` and confirmed by Python.

**Result:**

| Stage | Length | Abel (mod 5) |
|-------|-------:|:------------:|
| Original [c12,c9] | 28,652 | (0,0) ✓ |
| Lifted {a,b,m} | 544 | (0,0) ✓ |
| Coreless {a,b} | 15,328 | (0,0) ✓ |
| After power reduction | 10,624 | (0,0) ✓ |
| After fixed braid | 8,402 | (0,0) ✓ |
| After Python KB rules | 8,394 | (0,0) ✓ |
| After Rust braid_reduce | 8,388 | (0,0) ✓ |
| **final_beam.txt** | **7,245** | **(0,0) ✓** |

**What this establishes:** The 7,245-char word and the 28,652-char word are in the same fiber of the abelianization map. This is a *necessary* condition for equality in B(2,5). The previous invalid results (word_3707, word_3732, word_4882) all had abelianization (4,4) — failing this check.

**Limitation:** Necessary, not sufficient. Many distinct elements of B(2,5) have abelianization (0,0). Passing this check does not establish equality.

---

### Check 3 — KB rule `m → m_pure` exists in lifted KB system

**What was checked:** Grepped for `^m->` in `runs/b25_lifted/20260330_132357/kb_rpo/input.kbprog.live`.

**Result:** Rule found: `m → [156-char all-lowercase word]`. The RHS matches `m_pure` as referenced in `ui/comm_12_9_reduced_data.json`.

**What this establishes:** IF the KB run in `runs/b25_lifted/20260330_132357/kb_rpo/` was performed on the correct B(2,5) lifted presentation, then this rule is a sound B(2,5) identity, proving `m = m_pure` in B(2,5), which validates the coreless expansion step.

**Limitation:** We have not independently verified that kbprog run used the correct presentation. The rule's existence is evidence, not independent proof.

---

## What has NOT been checked

1. **Reduction chain soundness** — each step from 28,652 to 7,245 has not been independently verified to preserve group equality. We have checked that each step type (KB rules, power reduction, braid reduction) is *claimed* to be sound, but have not re-derived or independently executed the chain.

2. **Rust braid_reduce and beam search correctness** — the Rust binary is claimed correct (uses `invert_word` / complement+invert approach, not the buggy Python inverse order), but this has not been independently verified by us.

3. **Fixed braid reduction code** — the Python fix has been reviewed (README documents the before/after), but has not been independently re-run on the intermediate words.

4. **m_standard = m_pure independent derivation** — we found the KB rule, but have not independently re-derived this equality.

5. **Sub-claim A residual gap** — we have not independently computed `[c12,c9]` as a word in the free group and compared to the 28,652-char string.

## Verdict

**#status/conjectured**

The evidence is consistent with the claim being true. The checks performed rule out the specific class of bug that invalidated the previous results (abelianization shift from braid_reduce_fast). The input word's identity with the Havas relator gives high confidence in the starting point. But the central question — does the reduction chain preserve group equality at every step? — has not been answered by independent means. Without GAP or a convergent rewriting system, we cannot close this claim.

### What would upgrade this verdict

- **To #status/replicated**: Re-run the corrected pipeline independently from scratch on the 28,652-char word, verify 7,245-char output matches, and verify the kbprog run's input presentation. No new tools needed.
- **To #status/proven**: Install GAP. Verify `m_standard = m_pure` via GAP in a B(2,5) quotient. Verify the full equality chain in a quotient group. (Full B(2,5) coset enumeration is infeasible — order 5^34 — but small quotients suffice to catch errors.)
