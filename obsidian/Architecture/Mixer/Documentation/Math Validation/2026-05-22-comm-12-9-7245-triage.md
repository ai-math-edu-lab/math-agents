# Triage — B(2,5) comm_12_9 7,245-char reduction

## The Claims, Separated

The master claim decomposes into three independent sub-claims:

### Sub-claim A: The 28,652-char input word IS [c12, c9] in B(2,5)
**Status: evidenced** (but not formally proven)

The 28,652-char word appears verbatim in `b25_relators_havas_verified.txt` at entry `comm_12_9`. This file contains 153 relators from the Hall–Sims–Havas computer verification of B(2,5)'s finiteness. Entry comm_12_9 is labelled as the commutator [c12, c9]. Exact-string match confirmed by Python script.

What this establishes: the 28,652-char word is the same *string* that the Havas verification identified as comm_12_9. It does not prove that string equals the mathematical commutator [c12, c9] without also trusting that the Havas relators file was generated correctly — which is a reasonable trust assumption but not a formal proof.

**Residual gap:** We have not independently computed [c12, c9] from first principles and verified it matches the 28,652-char string. Could be verified in GAP: `c12 := Comm(Comm(a,b), a^11*b*a^-11); c9 := ...; c12*c9^-1` in the free group, then check the word.

### Sub-claim B: The reduction chain 28,652 → 7,245 preserves group equality in B(2,5)
**Status: partially evidenced, NOT verified**

This is the load-bearing sub-claim. It requires every one of the following 7 steps to be a sound equality-preserving operation in B(2,5):

| Step | Operation | Soundness basis | Verified? |
|------|-----------|-----------------|-----------|
| 28652 → 544 | Lifting: b → m where m = X⁻¹bX | m defined by the lifted B(2,5) presentation | Conditional on m_standard = m_pure (see step 3) |
| 544 → 544 | KB reduction in lifted {a,b,m} space | KB rules are sound by construction (if KB ran on correct presentation) | Not independently verified |
| 544 → 15328 | Coreless expansion: m → m_pure | Requires m_standard = m_pure in B(2,5) | **KEY GAP** — see below |
| 15328 → 10624 | Power reduction (w^5 = 1) | Trivially valid in B(2,5) by exponent-5 law | ✓ Sound |
| 10624 → 8402 | Fixed braid reduction | Sound IF the fixed Python code is correct | Code fix documented; not independently re-run |
| 8402 → 8388 | Rust braid_reduce --no-beam | Rust implementation claimed correct | Not independently verified |
| 8388 → 7245 | Rust beam search with KB rule bank | Uses KB rules as equalities | Not independently verified |

**KEY GAP — m_standard = m_pure:**
The expansion step uses `m_pure` (156-char all-lowercase word) as the expansion target, but `m_standard` (71-char mixed-case word `X⁻¹bX`) is the definition. For the expansion to be valid, these two must be equal in B(2,5).

Evidence found: the rule `m → m_pure` appears in `runs/b25_lifted/20260330_132357/kb_rpo/input.kbprog.live`. If that KB run was performed on the correct B(2,5) lifted presentation, this rule IS a valid B(2,5) identity. But we have not verified that the kbprog.live file was generated from the correct presentation, nor re-derived the rule independently.

### Sub-claim C: The 7,245-char word equals [c12, c9] in B(2,5) (compound claim)
**Status: conjectured**

This is the conjunction of A and B. Since B is not verified, C is not verified. Abelianization preservation is necessary but not sufficient: a word with abelianization (0,0) that does NOT equal [c12,c9] would pass all abelianization checks.

## What Abelianization Preservation Does and Does NOT Prove

Abelianization preservation establishes: the 7,245-char word and the 28,652-char word map to the same element of Z/5 × Z/5 under the abelianization homomorphism. This is a *necessary* condition for equality in B(2,5). It is NOT sufficient: there exist many distinct elements of B(2,5) with abelianization (0,0) (the commutator subgroup is large). The abelianization check is best understood as a bug-detection tool (it caught the 3707/3732/4882 errors), not a verification tool.

---

## What Remains Unverified

1. That the reduction chain preserves group equality at every step (Sub-claim B entire)
2. That `m_standard = m_pure` in B(2,5) (via independent means — the KB rule is evidence but not independent verification)
3. That the Rust braid_reduce and beam search steps are sound (they are claimed correct but not independently verified here)
4. That the 28,652-char string was correctly computed as [c12,c9] from the group presentation (Sub-claim A residual gap)

## Escalation Required

Full verification of the claim requires GAP. Specifically:
- To close Sub-claim A: `Comm(c12, c9)` in the free group, compared to the 28,652-char string
- To verify `m_standard = m_pure`: `IsIdenticalObj(m_standard, m_pure)` in a B(2,5) quotient, or re-deriving the lifted KB rule
- To close Sub-claim C (full equality): requires either a convergent rewriting system (infeasible with current tools) or GAP coset enumeration (infeasible for full B(2,5) — order 5^34 — but feasible in smaller quotients)
