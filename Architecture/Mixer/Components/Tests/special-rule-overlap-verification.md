# Special-rule overlap verification

## What it is

A static analysis tool that reads a rule bank and a target word, identifies "special" rules (those whose RHS is a substring of the target word), and classifies them by four overlap types between the rule's LHS and the target word: (1) LHS is a full substring, (2) LHS has a prefix overlap, (3) LHS has a suffix overlap, (4) LHS has no overlap. It also counts how many special rules are directly applicable (LHS is a substring) vs. those that are "nearby" but cannot fire yet.

## Why it exists

When kbprog biases toward rules whose RHS overlaps with the target word, a key question is: can those rules actually fire on the current word, or are they only "nearby"? A rule where LHS is not a substring of the target cannot reduce it directly; it only becomes applicable if an earlier reduction produces the pattern the LHS matches. This test separates rules that can act now from those that are structurally related but blocked — informing decisions about whether a particular agent is generating useful rules or just related-but-unapplicable ones.

## How it fits

Run on a rule bank generated by a biased kbprog agent, against the same target word used to bias it:

```
run_biased_agents.sh → input.kbprog.live
→ verify_special_rules.py <target_word> <input.kbprog.live> [--verbose]
  → reports: % special rules, 4 overlap type counts, applicable vs. blocked
```

## Critical invariants — why each one exists

### special-definition

`verify_special_rules.py` defines special as "RHS is a substring of the target word" — not the k-gram criterion. This is the intuition: a rule `LHS → RHS` is "special" if its RHS, the result of applying it, is a word that appears inside the target. Such a rule, if applicable, moves the word toward containing more of the target structure. The k-gram criterion ([[kgram-overlap-verification]]) is a proxy for this that kbmag can evaluate cheaply without doing full substring search. The two criteria are related but not identical — a rule can be k-gram-special without being RHS-special, and vice versa for rules with short RHS.

## Public surface — script interface

```
python tests/verify_special_rules.py <special_word_file> <rules_file> [--verbose]
```

- `special_word_file` — first non-comment line is the target word
- `rules_file` — `.kbprog.live` format
- `--verbose` — print detail for each of the first 50 special rules

**Output:**
- Count of special rules (RHS-is-substring) vs. non-special
- For special rules: count with LHS-is-substring, LHS-has-prefix-overlap, LHS-has-suffix-overlap, LHS-has-no-overlap
- Directly applicable special rules (LHS in word) with position context
- For comparison: what fraction of *non*-special rules are applicable to the word
